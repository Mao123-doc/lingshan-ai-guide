🏗 缺陷 #1 方案：数字人升级 — 大尺寸 + 真 viseme + 情感联动

❌ 现状

  • 画布 300×400 → 实际显示约 200px 宽
  • PixiJS 初始化慢 → 2 秒后切 fallback 静态图
  • viseme 口型叠加在 fallback 上不生效
  • 情感变化只改 drop-shadow 颜色

✅ 方案

  • 尺寸翻倍：画布 500×660 → 占页面 40% 高度
  • 纯 DOM 动画：不用 PixiJS（初始化太慢），用 CSS transform + requestAnimationFrame
  • 口型叠加层：AI 肖像上叠加 6 个 CSS 嘴型 div，viseme 时间轴驱动切换
  • 情感系统：眼部裁切 + 眉毛旋转 + 腮红透明度 + 整体缩放
  • 预加载：页面加载时创建 Image 对象，不等待 PixiJS

🔊 缺陷 #2 方案：流式 TTS — 文字到语音零延迟

🔄 状态流转：旧 vs 新

❌ 旧流程（串行）

用户发送→ 等待全文
(2-4s)→ 全文到达→ 调 TTS
(7-10s)→ 播放语音

总延迟：文字 3s + TTS 8s = 11s

✅ 新流程（并行流水线）

用户发送→ 首个 token
(0.3s)
流水线 A
token 陆续到达
→ 前端逐字显示
流水线 B（并行）
攒够 20 字 → 立即调 TTS
→ 边生成边播放

总延迟：文字 0.3s + 首段 TTS 1.5s ≈ 2s

📡 流式 TTS 数据流

LLM stream token → SSE chunk → 前端缓冲
                              ├─→ 逐字渲染到气泡（即时）
                              └─→ 累积到 ≥20 字 → POST /tts → Python edge_tts → 音频返回 → 播放
                                  同时继续累积下一段 → 段段衔接播放

裁剪算法

输入：兴趣标签 + 时长上限 T
1. 按兴趣匹配路线模板（2h/4h/6h）
2. 如果 T < 模板时长 → 贪心裁剪：按景点评分排序，依次加入直到累计时长 ≤ T
3. 如果 T > 最长模板 → 推荐深度体验 + "额外探索"区域
4. 每个路线输出：景点列表 + 预估时长 + 步行距离 + 最佳出发时间

🖥 缺陷 #5 方案：管理后台数字人预览修复

数字人形象预览（修复）

  • 独立渲染 → 不受配置保存影响
  • 实时反映服装/配色变更

语音预览（修复）

  • 绑定 voice_id 选择 → 调用 /tts API
  • 用选中语音朗读示例文本："你好，我是灵小禅"
  • 播放按钮 → 加载中状态 → 播放 → 可暂停

🧪 测试策略 & 边界情况

数字人边界

  • 图片加载失败 → 显示占位插画
  • viseme 数组为空 → 嘴型保持"闲"
  • 快速切换情感 → 0.3s 过渡动画防抖
  • 浏览器不支持 WebGL → 纯 DOM fallback

TTS 边界

  • TTS 服务不可用 → 浏览器 SpeechSynthesis
  • 文本 < 2 字 → 不触发 TTS
  • 段落衔接 → 每段末尾 300ms 交叉淡化
  • 用户打断 → 立即停止当前播放+清空队列

路线边界

  • 时长为 0 → 默认推荐 2h 路线
  • 兴趣无匹配 → 回退"文化"路线
  • 路线景点全部关闭 → 提示联系客服
