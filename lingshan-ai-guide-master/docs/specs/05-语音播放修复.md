# 语音播放功能修复 — 设计文档

## 问题诊断

三个 bug 的根因分析：

### 问题 1：点击播放无反应
**根因**：`handlePlayClick` 依赖 `activePlayingMsgId` 状态来判断"此消息是否正在播放"。但 `stopPlayback()` 内部调用 `setActivePlayingMsgId(null)`，React 的状态更新是异步的。在点击→stopPlayback→fetchTTS→播放的链路中，`activePlayingMsgId` 在 fetchTTS 完成后可能还是旧值，导致 `setActivePlayingMsgId(msgId)` 后的状态不一致。更关键的是：`handlePlayClick` 是一个 `useCallback`，闭包捕获了旧版 `activePlayingMsgId`，内部逻辑可能走到错误分支。

### 问题 2：部分消息缺按钮
**根因**：`showBtn = isAi && hasContent && !isStreamingMsg` 条件太宽。但 `preCacheTTS` 只在 SSE `done` 事件和 fallback 路径中调用。当 AI 回复通过 `type: 'done'` 结束时调用 `preCacheTTS`，但如果 SSE 流在 done 之前就结束（比如网络问题），`preCacheTTS` 不会被调用，消息的 `ttsStatus` 保持 `undefined`。此时 `btnStatus` 默认为 `ready`（`|| 'ready'`），按钮会显示但无缓存音频。点击后进入 fetchTTS 分支，如果 TTS 失败，`audioB64` 为 null，走"静默口型"分支，看起来像"没反应"。

### 问题 3：语音只播一段就停
**根因 A**：`visemeTimerRef.current` 的 setInterval 中，当 `elapsed > maxTime` 时执行 `clearInterval` 但没有调用 `stopPlayback()` 的完整清理逻辑，audio 可能继续播放但 viseme 停了。不过这不会导致 audio 提前停止。

**根因 B（更可能）**：React 组件重渲染导致 `useEffect` cleanup 被触发。QAPage 的 auto-scroll useEffect 监听了 `messages` 数组：
```ts
useEffect(() => {
  const el = messagesContainerRef.current;
  if (el) el.scrollTo({ top: el.scrollHeight, behavior: 'smooth' });
}, [messages]);
```
这不影响 audio。但关键在：`handlePlayClick` 中有 `setActivePlayingMsgId` 和 `setIsSpeaking`，每次调用都触发重渲染。重渲染不会中断 audio。

**根因 C（最可能）**：`stopPlayback` 在不同地方被调用。检查代码：
- `handleSend` 开头调用 `stopPlayback()` —— 如果用户在播放时发新消息
- `handleClearChat` 调用 `stopPlayback()`
- `handlePlayClick` 内部调用 `stopPlayback()`

当用户点击播放按钮时，React 的 `onClick` handler 执行 `handlePlayClick`。但在函数组件中，`handlePlayClick` 是一个 `useCallback`，它被 `activePlayingMsgId` 等 deps 重新创建。如果在 `stopPlayback()` 中 `setActivePlayingMsgId(null)` 后，组件重渲染，新渲染可能触发某些 effect 清理掉 audio。

**根本解**：audio 播放应该完全由 ref 管理，不受 React 渲染周期影响。

## 修复方案

### 方案：Ref 驱动的音频管理

**核心思路**：音频播放逻辑全部通过 ref 管理，不受 React 状态变化影响。

1. `audioRef` 管理当前 audio 元素，在 stopPlayback 中只操作 ref
2. `isCurrentlyPlaying(msgId)` 的判断也通过 ref 而非 state
3. `stopPlayback` 纯粹操作 ref + DOM，不触发状态更新（除非必要）
4. 移除 `activePlayingMsgId` state，改用 `playingMsgIdRef`
5. 修复按钮显示逻辑：`preCacheTTS` 始终被调用，默认显示按钮，点击时再 fetch

### 具体修改

1. `activePlayingMsgId` state → `playingMsgIdRef` ref
2. `isSpeaking` state → 保留（用于驱动数字人 CSS 动画）
3. 移除所有 `setActivePlayingMsgId`、`setIsSpeaking` 的冗余调用
4. `stopPlayback` 精简为纯 ref 操作
5. 所有 AI 消息始终显示播放按钮
6. 点击按钮时：停止当前 → 取缓存或 fetch → 播放 audio + viseme → audio.onended 清理

## 实现步骤

1. 改 state → ref：`playingMsgIdRef` 代替 `activePlayingMsgId`
2. 重写 `stopPlayback`：仅操作 ref，不 setState
3. 重写 `handlePlayClick`：用 ref 判断状态，消除闭包陈旧问题
4. 简化按钮渲染：始终显示，用 ref 判断 playing 态
5. 验证音频完整播放不中断
