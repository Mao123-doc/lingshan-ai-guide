# 灵山胜境AI数字人 — 竞赛级改造方案

## 问题陈述

项目骨架完整（RAG管线、管理后台、数据架构），但三个致命短板导致竞赛评分在62-68分：
1. "数字人"只是 `<img>` 标签 + CSS，Live2D SDK闲置（-10分）
2. 语音交互是"带播报的聊天机器人"而非自然对话（-5分）
3. 后端碎片化，评委可能跑不起来（-3分）

目标：**65 → 85-88分**（一周务实路线）

## 约束条件

- 时间：约1周
- 策略：增量改进，不完全重写
- 稳定性优先：每个改造独立可回退
- 利用现有基础设施（Live2D SDK已有、viseme管道已有、WebSocket已预留）

---

## 改造方案（按优先级排序）

### Phase 1: 🔴 Live2D 数字人真正动起来（最高优先级 — 影响15分）

**现状**：`DigitalHuman.tsx` 渲染 `<img>` + CSS @keyframes，SDK和模型闲置

**改造方案**：新建 `Live2DCanvas.tsx` 组件，使用 Live2D Cubism 4 SDK 在 WebGL Canvas 上渲染 hibiki 模型

```
Live2DCanvas.tsx
├── 初始化：加载 .moc → 创建 Model → 绑定纹理 → 启动渲染循环
├── 口型同步：viseme_id → PARAM_MOUTH_OPEN_Y 映射
├── 表情切换：emotion → expression .exp.json 加载
├── 空闲动作：随机播放 idle .mtn 文件
├── 物理模拟：加载 hibiki.physics.json
├── 呼吸效果：PARAM_BREATH 正弦波
└── 眼球追踪：PARAM_EYE_BALL_X/Y 随机微动
```

**关键设计决策**：
- 不重写 DigitalHuman.tsx，而是新增 Live2DCanvas 作为替代渲染层
- DigitalHuman 组件增加 `useLive2D` prop，渐进切换
- 如果 WebGL 不可用，自动回退到现有静态图片模式
- 复用现有的 viseme 数据管道（`tts_viseme.py` 已经生成了 viseme_id/timing）

**口型映射**（利用已有 viseme 数据）：
```
viseme_id 0 (rest) → PARAM_MOUTH_OPEN_Y = 0
viseme_id 1 (aa)   → PARAM_MOUTH_OPEN_Y = 1.0
viseme_id 2 (e)    → PARAM_MOUTH_OPEN_Y = 0.5
viseme_id 3 (i)    → PARAM_MOUTH_OPEN_Y = 0.2
viseme_id 4 (o)    → PARAM_MOUTH_OPEN_Y = 0.7
viseme_id 5 (u)    → PARAM_MOUTH_OPEN_Y = 0.3
viseme_id 6 (cd)   → PARAM_MOUTH_OPEN_Y = 0
```

**风险**：Live2D Web SDK 较复杂。**缓解**：先实现最小可用版本（加载模型+口型同步），其他功能渐进叠加。

### Phase 2: 🔴 语音对话闭环（高优先级 — 影响体验分）

**现状**：打字→等文字→手动点播放按钮→听语音。流程断裂。

**改造方案**：
1. AI回答到达后**自动播放** TTS 语音（不用手动点击播放按钮）
2. 播放完毕后**自动开启麦克风**等待下一轮语音输入（可选开关）
3. 数字人说话时展示"聆听中/思考中/讲解中"三种状态

**关键改造点**：
- `QAPage.tsx`：添加 `autoPlay` 状态，回答到达→自动调用 handlePlayClick
- 保留手动播放按钮作为 fallback
- 添加"连续对话模式"toggle
- 优化 Web Speech API 的中文识别体验（添加 interim results 显示）

---

### Phase 3: 🟡 后端统一编排（中优先级 — 影响演示体验）

**现状**：三个独立服务，各自手动启动

**改造方案**：
1. 编写统一的 `docker-compose.yml`（推荐）或启动脚本
2. Node.js 主服务启动时自动检查 TTS/Vector 服务健康状态
3. 添加启动进度日志

---

### Phase 4: 🟡 RAG 增强（中优先级 — 影响准确率）

**现状**：三层检索（向量→结构化→关键词），但无 Reranker

**改造方案**：
1. 添加轻量级 Reranker：用 LLM API 对 top-5 结果做重排序
2. 添加 query rewriting：识别口语化查询 → 转为搜索友好格式
3. 增加知识库覆盖（补充现有的 data/raw 文件内容）

**Query Rewriting 示例**：
```
"那个大佛到底有多高" → "灵山大佛高度参数"
"有没有不要钱的地方" → "免费景点 免票"
```

---

### Phase 5: 🟡 GPS/位置感知（中优先级 — 体现行业实用）

**现状**：无位置功能

**改造方案**：
1. 游客端首页添加"📍 当前位置"按钮
2. 使用浏览器 Geolocation API 获取经纬度
3. 匹配最近景点（利用 visitor.ts 中已有的 ALL_SPOTS 经纬度数据）
4. 生成到达路线提示
5. 兜底：提供手动选择"我在哪个景点"的列表

---

### Phase 6: 🟢 打磨优化（低优先级 — 锦上添花）

1. 修复管理后台满意度数据硬编码（`getSatisfactionTrend` 返回0时默认0而非4.5）
2. 添加 PWA manifest + service worker
3. 移动端响应式适配
4. 游客端添加满意度评分组件（后端API已就绪）
5. 知识库管理页集成在线准确率测试（已有 test_accuracy.py）
