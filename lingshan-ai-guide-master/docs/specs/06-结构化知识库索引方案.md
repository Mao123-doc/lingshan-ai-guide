# 结构化知识库索引方案 — 设计文档

## 问题陈述

当前分块 + 关键词匹配方案在搜索"灵山大佛"时会匹配到无关景点，且分块内容是半截话，丢失上下文。

## 核心洞察

知识库数据天生就是结构化的：
- 22 个景点 × 10 个统一字段（景点ID、名称、位置、建筑参数、核心功能、文化内涵、详细介绍、游玩亮点、开放信息、备注）
- 每个字段的值都是完整、自包含的自然语言描述
- 不需要分块！每个字段天然就是一个检索单元

## 推荐方案：结构化建模 + 两级检索

### 数据建模

解析 docx/txt → 22 个 `ScenicSpot` 对象，每个包含：
```
{
  id: "LS-011",
  name: "灵山大佛",
  area: "灵山胜境",
  fields: {
    "位置": "祥符禅寺北侧，秦履峰南侧",
    "建筑参数": "通高88米（佛体79米，莲花瓣9米），总用铜量725吨...",
    "核心功能": "祈福朝圣、地标景观",
    "文化内涵": "五方五佛之东方大佛，佛教文化象征...",
    "详细介绍": "灵山大佛是世界最高露天青铜释迦牟尼立像...",
    "游玩亮点": "登顶抱佛脚，俯瞰太湖全景",
    "开放信息": "8:00-17:00",
    "备注": "重要佛教朝圣场所"
  }
}
```

每个字段值作为一条检索"文档"，自带完整上下文：
`【灵山大佛】建筑/景观参数：通高88米（佛体79米，莲花瓣9米），总用铜量725吨...`

### 两级检索

**第一级（景点匹配）**：用户查询 → 分词 → 匹配景点名
- "灵山大佛有多高" → 匹配 "灵山大佛"
- "九龙灌浴几点表演" → 匹配 "九龙灌浴"
- 支持模糊匹配：查询中的子串匹配景点名的子串

**第二级（字段匹配）**：在匹配到的景点内，找到最相关的字段
- 关键词命中 "高" → 匹配 "建筑参数"（含"88米"）
- 关键词命中 "表演" → 匹配 "开放信息"（含"表演时间"）
- 没有明确字段关键词 → 返回"详细介绍"作为默认

### 索引结构

```
[
  { id: "LS-011_building",  spot: "灵山大佛", field: "建筑参数", text: "通高88米..." },
  { id: "LS-011_culture",   spot: "灵山大佛", field: "文化内涵", text: "五方五佛..." },
  { id: "LS-006_show",      spot: "九龙灌浴", field: "演艺信息", text: "大型音乐动态群雕..." },
  ...
]
```

### 优势

1. **零串景点**：景点名是第一级过滤，绝对不跨景点匹配
2. **无半截话**：每个文档是一个完整的字段值（由原始 docx 的字段边界保证）
3. **快**：不需要向量计算，全内存 O(n) 关键词匹配
4. **可解释**：返回结果标注 `【景点名】字段名`，管理员一眼看懂
5. **可扩展**：管理员上传新 docx 时自动解析字段并入索引
